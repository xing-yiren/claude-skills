# 小红书技术博客模板（technical-blog-generator-v2）

> 使用说明：
> - 仅填写有证据支撑的内容；无依据处请写"原文未给出"。
> - 避免夸张词和绝对化结论。
> - 不要把所有题材都写成同一种“技术博客”；先判断类型，再选模板。
> - 深度技术帖推荐 600-900 字；轻量发现帖 150-400 字也可。

## 通用规则

### 标题（1 个，≤20 字）

`[一句完整话，直接说清发现了什么、差异在哪，或为什么值得看]`

标题规则：
- 必须是一个可单独成立的完整句，不能用"最近了解到一个项目"这类泛开场充当标题
- 如果原文属于发布/上线/开源/推出类内容，显式包含"发布""上线""开源"等事件词
- 避免公式化反差句式，如"不是 X 而是 Y"——能直接陈述就优先直接陈述
- 优先使用"一句话创建""刚发布""都更新了啥"这类强信息词，但情绪词必须服务于正文核心机制
- 要有信息量但不能像标题党，准确反映内容

### 正文结构

动笔前先定"一句话主旨"，用它检验每一段是否服务同一个重心。

```
1. 开头：一句话交代这篇只讲一个什么判断或发现，直接给核心认知
2. 中段：3-5 个短段落，每段 2-3 句，用 emoji 或编号做视觉分隔
3. 结尾：一句回扣开头主旨的 takeaway + 3-5 个话题标签
```

### 文风约束

- 采用用户/开发者第一视角，分享感强，像一个发现了好东西忍不住分享的朋友
- 短句为主，单句不超过 25 字
- 口语化但不水，避免"绝绝子""YYDS""太酷啦"等空洞感叹
- 主动回避"不是……而是……""重点来了""真正拉开差距的是……"等公式化转折句
- 技术术语保留，但紧跟一句大白话解释（"即……""简单说就是……"）
- 不做虚假"亲身经历""我朋友""我试了"等编造叙事——如果原文是别人的案例，就说"有人试了""案例里"
- 用具体场景代替抽象描述：不说"效率很高"，说"以前要手动配半天，现在一句话搞定"
- 结构上要跟原文的事实递进顺序一致，但表达上仍然要保留小红书的朋友分享感

### 实践/案例类稿件额外要求

除了讲创建过程和协作机制，必须补一段"展示效果"：读者看完需要知道屏幕上具体看到了什么、为什么值得发。

允许的展示效果（必须回源可证）：
- 生成出的角色/配置/结构长什么样
- 运行时过程有哪些节点可见
- 最终输出的结果形态
- 与手动操作相比省掉了哪一步

禁止只讲机制对不对，不讲最后看到了什么。

### 话题标签

3-5 个，格式 `#标签名`，放在正文末尾。

组合策略：产品名 + 核心概念在前（精准搜索），领域泛标签在后（泛流量）。

示例：`#TeamSkills #JiuwenClaw #openJiuwen #多Agent #开源项目`

### 宣传链接

如果来源文章带有宣传/发布属性，正文末尾（标签前）补入宣传链接：
- 链接来源：来自台账关键词清单中的关键平台/资源项
- 每篇至少覆盖一条链接；如果正文未提及 Hub，则同时补 Hub 地址并加一句简短介绍
- 链接必须来自来源文章明确给出的地址，不编造

## 先判断帖子类型，再选对应模板

### A. 教程清单类模板

适用于：技巧合集、最佳实践、使用建议。

**推荐结构**：
- 开头：一句话痛点 + 一句话收益
- 中段：技巧列表（建议 5-10 条），允许使用 ❶❷❸ 或 1/2/3
- 结尾：一句“先从哪一条开始试”的收束

**示例骨架**：
- 痛点：`[刚上手总不顺/总踩坑的问题]`
- 收益：`[学会后会带来什么变化]`
- 技巧列表：
  - `❶ [技巧1]`
  - `❷ [技巧2]`
  - `❸ [技巧3]`
- 收束：`[建议先从哪一条开始用]`

**配图建议**：
- 封面：大标题 + 品牌元素 + 收益点
- 内页：分页技巧卡，每页 1-2 条技巧

### B. 实践体验类模板

适用于：产品试玩、真实上手、值不值得、demo 体验。

**推荐结构**：
- 开头：个人/朋友切口 + 使用场景
- 中段：痛点 → 怎么用 → 看到的效果
- 结尾：适合谁 / 值不值 / 会不会继续用

**示例骨架**：
- 场景：`[我为什么开始用/朋友为什么拿给我看]`
- 痛点：`[原来最烦的点是什么]`
- 上手：`[具体怎么用]`
- 效果：`[屏幕上看到了什么/结果有什么变化]`
- 结论：`[它更适合谁]`

**配图建议**：
- 优先真实界面截图、结果截图、前后对比截图
- 次选流程图或概念封面
- 禁止只给抽象配图，不给使用结果

### C. 技术解读类模板

适用于：论文解读、版本更新拆解、产品能力分析。

**推荐结构**：
- 开头：为什么这次值得讲
- 中段：关键变化 / 核心点 3-6 条
- 结尾：这次更新最值得关注的影响是什么

**示例骨架**：
- 引子：`[为什么今天值得看它]`
- 重点：
  - `1. [关键变化1]`
  - `2. [关键变化2]`
  - `3. [关键变化3]`
- 影响：`[这对开发者/用户意味着什么]`

**配图建议**：
- 封面：大标题直给主题
- 内页：图表、结论卡、对比卡
- 若引用技术报告，优先保留关键表格/图表

### D. 科普概念类模板

适用于：新概念解释、范式演进、术语扫盲。

**推荐结构**：
- 开头：为什么最近总被问 / 为什么今天值得讲
- 中段：概念是什么 → 跟旧概念的关系 → 为什么现在重要
- 结尾：一句带走的认知

**示例骨架**：
- 引子：`[最近为什么老看到/老被问这个词]`
- 定义：`[概念到底是什么]`
- 对比：`[它和旧概念有什么关系/区别]`
- 结论：`[为什么现在重要]`

**配图建议**：
- 封面：问题句或大标题
- 内页：分页解释卡、关系图、对比卡

### E. 新闻 / 活动类模板

适用于：功能更新、活动预告、发布快讯。

**推荐结构**：
- 开头：一句话消息
- 中段：亮点 bullet
- 结尾：CTA / 时间地点 / 参与方式

**示例骨架**：
- 消息：`[这次发生了什么]`
- 亮点：
  - `- [亮点1]`
  - `- [亮点2]`
  - `- [亮点3]`
- CTA：`[怎么参加/去哪里看/什么时候开始]`

**配图建议**：
- 优先官方海报或封面图
- 正文不强制长文，更重要的是亮点和报名/参与信息

### F. 深度技术博客子模板

仅当用户明确想做长文式技术拆解时，才使用下面这个结构：

#### 1) 技术背景及解决的问题
- 当前痛点：`[问题1]`、`[问题2]`
- 影响范围：`[性能/稳定性/开发效率/维护成本]`
- 本文要解决：`[目标问题]`

#### 2) 技术方案（方案介绍）
- 总体思路：`[一句话架构策略]`
- 关键模块：
  - `[模块A]`：`[职责]`
  - `[模块B]`：`[职责]`
  - `[模块C]`：`[职责]`
- 关键机制：`[流程、策略或算法要点]`

#### 3) 技术方案（原理图）
- 若原文有图：`[引用原文图并注明来源]`
- 若原文无图：`[插入 Mermaid 原理图代码块或渲染结果]`

#### 4) 效果收益（必须可验证）
- 指标/现象 1：`[原文或代码可验证事实]`
- 指标/现象 2：`[原文或代码可验证事实]`
- 指标/现象 3：`[原文或代码可验证事实]`

> 禁止捏造 benchmark、用户规模、线上故障收益等数据。

#### 5) 如何使用
1. `[前置条件]`
2. `[安装或准备步骤]`
3. `[核心使用步骤]`
4. `[注意事项]`

#### 6) 如何启动
```bash
# 仅填写已验证命令
[启动命令1]
[启动命令2]
```

#### 7) demo case
- demo 场景：`[输入条件]`
- demo 过程：`[关键步骤]`
- demo 结果：`[可观察输出]`
