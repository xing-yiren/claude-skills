# article-atomize

## 概述

把一篇长文拆成多条可独立发布的短内容，每条只讲一个重点。支持指定目标平台（知乎 / 小红书 / 双平台），所有输出必须能回溯到原文明确依据。

## 输入要求

用户需要提供以下信息（缺少任何一项，先向用户确认）：

1. **原始长文**：全文文本或可访问的链接。注意：微信公众号等部分平台的链接可能因反爬机制无法通过 WebFetch 或 curl 抓取，此时请用户直接将文章正文粘贴到对话中，或导出为本地 markdown/txt 文件后读取
2. **目标平台**：`知乎` / `小红书` / `两者都要`
3. **期望拆分数量**：如 5 / 8 / 10 条（可选，默认由模型根据内容密度判断）
4. **内容类型**（可选，帮助调整拆解策略）：
   - `知识型`：偏概念、原理、认知框架
   - `观点型`：偏立场、判断、反驳
   - `经验型`：偏实践、方法、教训
   - `方法型`：偏步骤、流程、工具
5. **风格强度**（可选，默认 `标准`）：
   - `保守`：尽量贴近原文表达，少做修辞扩展
   - `标准`：允许安全改写和适度口语化
   - `偏传播`：在事实边界内做更强的表达优化
6. **严格模式**（可选，默认开启）：
   - `开启`：禁止任何原文没有明确支持的推断
   - `关闭`：允许合理延伸（会标注风险等级）

## 执行流程

按以下步骤顺序执行，每步完成后给出结果，再进行下一步。

---

### 步骤 1：原文拆解 —— 建立证据台账

先不写任何短文。只做原文的深度拆解，形成结构化的"内容台账"。

输出格式：

```
## 证据台账

### 1. 核心论点
- 论点：xxx
- 原文依据：[第X段] "原句摘录..."
- 类型：观点
- 置信度：高

### 2. 支撑论据
- 论据：xxx
- 原文依据：[第X段] "原句摘录..."
- 类型：事实/数据/案例
- 置信度：高/中/低

### 3. 关键数据
- 数据：xxx
- 原文依据：[第X段] "原句摘录..."
- 类型：数据
- 置信度：高/中

### 4. 明确结论
- 结论：xxx
- 原文依据：[第X段] "原句摘录..."
- 类型：结论
- 置信度：高

### 5. 原文金句（可直接引用）
- "xxx"（第X段）
- "xxx"（第X段）

### 6. 原文中的不确定表述
- "xxx"（第X段）— 作者使用了"可能""有些情况下"等限定词，改写时必须保留

### 7. 关键词清单

从原文中提取以下三类关键词，后续所有输出（标题、正文、标签）必须覆盖：

**产品/项目名**：原文涉及的具名产品、项目、组织名。如 openJiuwen、JiuwenClaw、Claude Code
**核心概念/范式**：原文提出的专有概念或方法论。如 Team Skills、Coordination Engineering、Agent Team
**关键平台/资源**：原文给出的独立网址或平台名。如 Team Skills Hub（teamskills.openjiuwen.com）

### 8. 行动号召识别

从原文中识别所有推广性质的呼吁（CTA），单独列出：

- 号召内容：xxx
- 原文依据：[第X段] "原句摘录..."
- 类型：下载使用 / 参与共建 / 上传分享 / 关注社区 / 其他
- 适用选题：方向 #N

### 9. 内容密度评估
- 可拆分的内容点数量建议：X 个
- 理由：xxx
```

**规则**：
- 每条台账必须附原文引用，不允许凭印象归纳
- 置信度标准：高=原文明确陈述，中=原文暗示或可合理推断，低=仅有碎片信息
- 不确定表述必须单独列出，改写时作为约束条件
- 若文章带有明显宣传/发布属性，在拆解阶段必须同步抽取"发布来源/发布主体/配套发布内容"三项信息：谁发布、发布了什么、是否同步发布了配套工具或平台
- 对工具、平台、能力的职责边界要单独记录，避免在改写时把多个能力顺滑地合并成一个功能包

---

### 步骤 2：选题原子化

基于证据台账，将长文拆成多个"单点内容单元"。每个单元只回答一个问题或传递一个认知。

**拆分原则**：

1. **优先遵循原文结构**：以原文的章节、小标题、段落群为自然边界进行拆分，不要打散原文的叙事逻辑后自由重组。原文的章节划分通常是作者已经做好的"内容单元化"，直接复用即可。
2. **用户指定的方向优先**：如果用户明确给出了拆分方向（如"按 4 个方向"），以用户方向为准，每个方向对应一个选题。
3. **一个选题 = 一个独立认知**：拆分后每个选题能脱离上下文独立阅读，不依赖前后篇。
4. **宣传信息按需补足**：如果原文是发布/推广类文章，每个独立选题在不失真的前提下，应补足必要的发布来源和主体信息，避免读者只看到局部能力，却不知道这是 openJiuwen / JiuwenClaw 发布的整体动作

选题筛选标准：

- ✅ 能独立成篇（不需要其他上下文就能读懂）
- ✅ 原文证据足够（至少有 1 条高置信度依据）
- ✅ 与其他选题重复度低
- ✅ 对目标平台有表达价值

输出格式：

```
## 选题清单

### 选题 1：{主题名}
- 一句话核心观点：xxx
- 原文依据：台账条目 #N, #M
- 适合平台：知乎 ✅ / 小红书 ✅
- 证据充足度：充足 / 一般 / 偏弱
- 风险等级：🟢 低 / 🟡 中 / 🔴 高（需人工确认）

### 选题 2：...
...

### 选题筛选说明
- 共提取 X 个选题，覆盖原文 Y% 的核心内容
- 舍弃的选题及原因：xxx
```

**风险等级定义**：
- 🟢 低：原文明确陈述，改写空间大
- 🟡 中：原文有暗示但非明确陈述，需保留限定词
- 🔴 高：原文信息碎片化，需要较多拼接，强烈建议人工确认

---

### 步骤 3：平台化改写

对每个选题，按用户指定的平台生成对应版本。

#### 知乎版内容规范

**标题风格**：
- 问题式：`为什么 X 不是 Y？`
- 判断式：`关于 X，最重要的一件事`
- 反常识式：`大多数人理解错了 X`

**正文结构**：
```
1. 开头：一句话抛出核心结论（不铺垫，直接给）
2. 展开：2-4 个论据或解释，层层递进
3. 落点：一个更稳的收束句，避免绝对化
```

**风格要求**：
- 技术属性明确，措辞克制，避免口语感叹（"说白了""搞""挺""蛮""巨"等不出现在知乎版中）
- 保留逻辑链条，使用"因此""这意味着""其核心在于"等正式过渡
- 可以保留复杂性和 nuance，允许技术术语但不堆砌
- 少用强情绪词（"震惊""颠覆""彻底"）
- 允许保留原文的限定词（"在某些条件下""可能""多数情况下"）
- 不强行金句化
- 发布/公告类内容采用正式新闻稿口吻；解读/分析类内容可稍灵活但保持专业度
- 发布/推广类文章的各篇拆解稿，在保持主题聚焦的同时，应补足必要的发布来源、发布主体和配套发布内容，避免割裂上下文
- 避免"说白了""搞""挺""蛮""巨""超级"等口语词；不使用感叹号做情绪强化

**内容类型与风格映射**：

| 原文类型 | 知乎版风格 | 示例标题格式 |
|----------|-----------|-------------|
| 产品/技术发布 | 正式新闻稿口吻，客观陈述 | `XXX 发布 YYY：{一句话定位}` |
| 规范/设计解析 | 技术解读，偏架构分析 | `{核心设计}——XXX 的规范设计解析` |
| 实战/案例 | 流程拆解 + 关键差异分析 | `从 X 到 Y——XXX 的实战拆解` |
| 生态/社区 | 行业视角，强调闭环与趋势 | `XXX 上线：{领域}的开放市场` |

**字数建议**：300-800 字

---

#### 小红书版内容规范

**平台约束**：
- 小红书不支持 markdown 渲染，最终输出必须是可直接复制粘贴的纯文本
- 标题 ≤20 字，正文 ≤1000 字
- 不支持加粗、代码块、表格等格式，用 emoji 和符号替代排版
- 发布方式为图文笔记（封面图 + 纯文字正文 + 配图）或视频笔记

**叙事视角**：
- 采用用户/开发者第一视角，分享感强，不做官方通告口吻
- 像一个发现了好东西忍不住分享的朋友，而非机构账号
- 可以用的视角：刚发现一个项目、试用了一个功能、被某个案例震撼到、对比了两个东西后发现差异

**标题风格**（≤20 字）：
- 发现式：`多 Agent 协作终于不用每次都从零开始了`
- 反差式：`一个文件夹就是一个 AI 团队？长见识了`
- 结果式：`一句话创建 23 个 AI 医生，我看了两遍`
- 类比式：`多 Agent 协作也有"应用商店"了？`
- 发布式：`JiuwenClaw 刚发布 Team Skills：多 Agent 协作终于能复用经验了`
- 标题必须是一个可单独成立的完整句，不能用"最近了解到一个项目"这类泛开场充当标题
- 如果原文属于发布/上线/开源/推出类内容，标题应显式包含"发布""上线""开源"等事件词，强化发布感
- 标题要直接说清楚发现了什么/差异在哪里/结果有多强，不要把信息留到正文里
- 避免公式化反差句式，如"不是 X 而是 Y"——这种表达读起来偏模板感，能直接陈述就优先直接陈述
- 优先使用"一句话创建""刚发布"这类强信息词，但情绪词必须服务于正文核心机制，不能只剩"我看了两遍"这种弱信息尾巴
- 标题要有信息量但不能像标题党，准确反映内容

**正文结构**：
```
1. 开头：一句话交代这篇只讲一个什么判断或发现。直接给核心认知，不在铺陈中慢慢靠近
2. 中段：3-5 个短段落，每段 2-3 句，用 emoji 做视觉分隔（🔍 😤 🎯 📁 🤔 👇 等）。每一段都必须服务于同一条主线，不平行罗列互不相关的事实点
3. 结尾：一句回扣开头主旨的 takeaway + 3-5 个话题标签
```
- 正文动笔前，先在内部分配一句"一句话主旨"，用它来检验每一段是否服务同一个重心

**风格要求**：
- 短句为主，单句不超过 25 字
- 口语化但不水，避免"绝绝子""YYDS""太酷啦"等空洞感叹
- 主动回避"不是……而是……""重点来了""真正拉开差距的是……"等公式化转折句——这类句式的 AI 模板感太重。能直接说结论就直说，不为了制造反差而刻意铺"先否定再肯定"
- emoji 用于分段标记，每段 1-2 个，不要连续堆砌
- 技术术语保留，但紧跟一句大白话解释（"即……""简单说就是……"）
- 不做虚假"亲身经历""我朋友""我试了"等编造叙事——如果原文是别人的案例，就说"有人试了""案例里"
- 用具体场景代替抽象描述：不说"效率很高"，说"以前要手动配半天，现在一句话搞定"
- 如果某篇承担的是"结构解析"角色，正文里仍可适度补入"怎么创建/怎么上手"的信息，避免只剩静态结构说明，缺少用户关心的使用感

**实践/案例类稿件额外要求**（适用于图文实践稿和视频帖文案）：
- 除了讲创建过程和协作机制，必须补一段"展示效果"：读者看完这篇稿子，需要知道这件事最后呈现出了什么、屏幕上能看到什么、为什么值得发
- 允许的展示效果（必须回源可证）：
  - 生成出的角色/配置/结构长什么样
  - 运行时过程有哪些节点可见
  - 最终输出的结果形态
  - 与手动操作相比省掉了哪一步
- 禁止只讲机制对不对，不讲最后看到了什么

**话题标签**：
- 3-5 个，格式 `#标签名`，放在正文末尾
- 标签必须覆盖台账"关键词清单"中的产品名和核心概念，不可遗漏
- 组合策略：产品名 + 核心概念在前（精准搜索），领域泛标签在后（泛流量）
- 示例：`#TeamSkills #JiuwenClaw #openJiuwen #多Agent #开源项目`

**宣传链接规则**：
- 如果原文带有宣传/发布属性，所有相关选题的正文末尾（标签前）都应补入宣传链接，让读者可以直接访问
- 链接来源：来自台账"关键词清单"中的关键平台/资源项，包括项目地址（GitCode/GitHub）和独立平台网址（Hub 等）
- 覆盖策略：每篇稿子至少覆盖一条链接；如果正文已提及 Hub，则补 Hub 地址；如果正文未提及 Hub，则同时补 Hub 地址并加一句简短介绍（如"配套的 Team Skills Hub 也已上线，可以下载现成的团队技能，也可以上传自己的"）
- 规则：链接必须来自原文明确给出的地址，不编造；GitCode/GitHub/独立网址三者不混淆

**输出格式**：
- 每个选题输出一个独立的 `.md` 文件
- 文件内容仅为纯文本正文（可直接全选复制到小红书编辑器），无 markdown 预览、无分析标注
- 文本中需要分行的地方用空行分隔
- 如果原文带有宣传/发布属性，在相关选题的正文末尾或标签前补入项目地址和平台链接（如 GitCode/GitHub 地址、Hub 网址等），让读者可以直接访问；链接必须来自原文明确给出的地址，不编造

**字数建议**：150-400 字

---

#### 媒体资源处理

原文附带的图片和视频资源，在生成短文中按以下规范处理：

**图片**：
- 使用 markdown 图片语法：`![描述](相对路径)`
- 路径中的空格必须用 `%20` 替换：`../资源目录/Team%20Skills介绍.png`
- 在原文中有明确配图位置的，在对应段落处插入；原文无明确位置的，在相关章节标题下方插入
- 不新增原文没有的配图

**视频**：
- 若输出平台支持视频嵌入（如本地 markdown 预览），可使用 HTML `<video>` 标签：
  ```html
  <video src="../资源目录/%20视频文件名.mp4" controls width="100%"></video>
  ```
- 若输出平台不支持直接嵌视频（如小红书图文），实践向/案例向帖子默认仍应有配图，不建议做成纯文字长帖
- 此时优先策略不是自动抽帧，而是在成稿中明确写出“建议从哪个视频截哪一类画面”
- 如果视频截图效果不理想，再退回到 AI 生图方案，生成一张概念封面图补位
- 若用户输出的最终平台是小红书图文，默认不要直接把自动抽帧文件写进正文，除非用户明确确认截图效果可接受
- 如果用户在后续反馈中否定了某种媒体策略（如视频截图、AI 配图、某类封面风格），必须回查并清理所有正文中的残留引用与说明，不能只修改 skill 不修改成稿
- 小红书图文优先使用原文自带图片、明确的视频截图建议、独立封面图，或改写文案以避免强依赖视频截图
- 视频路径中的空格同样需要 `%20` 编码
- 在原文提及该视频的对应段落处嵌入，或在对应平台文案中单独生成视频帖文案

**AI 生成配图**（当原文无合适配图时）：
- 如用户提供了文生图 API 配置，可以为缺少配图的选题生成封面图
- API 配置由用户提供，不硬编码模型名和 endpoint。用户需提供：endpoint URL、model 名称、API key、请求格式说明
- 生成后存到 output 目录，并在正文中标注配图路径
- 图片规格建议 1024×1024 或 1440×1080（小红书竖版），适合直接上传

**通用规则**：
- 媒体文件路径始终使用相对路径，以输出文件所在目录为基准
- 中文文件名和目录名不做修改，仅在引用时对空格做 URL 编码

#### 双平台通用约束

无论哪个平台，以下规则必须遵守：

| 规则 | 说明 |
|------|------|
| 不补数据 | 原文没有的数字绝不新增 |
| 不补案例 | 原文没有的具体案例绝不编造 |
| 不扩因果 | 原文说"相关"不改写为"导致" |
| 不升确定性 | 原文说"可能"不改写为"一定" |
| 不转普遍性 | 原文说"在某些领域"不改写为"所有行业" |
| 保留限定词 | 原文的"有些""多数""倾向于"必须保留 |
| 网址准确 | 原文给出的链接、平台名、项目地址必须准确使用，GitCode/GitHub/独立网址不混淆 |
| 平台声称需验证 | 如文案声称"在 GitHub 发现""在 XX 平台看到"，必须先验证该内容在对应平台上确实存在（如确认 GitHub 仓库存在、Hub 网址真实可访问）。"朋友推荐""我试了一下"等叙事手法可用，不做硬性禁止，但涉及具体平台/网址的声称必须可验证 |

### 步骤 4：文本质量检视

在进入事实校验之前，先对每条生成的短文做一轮文本质量检查。这一步只关注语言本身，不关注事实准确性。

**检查项**：

| 检查项 | 说明 |
|--------|------|
| 语病 | 缺主语、缺谓语、搭配不当、句式杂糅、成分残缺 |
| 错别字 | 同音错字、形近错字、多字漏字 |
| 通顺度 | 句子是否拗口、是否有多余的"的""了""呢"等虚词堆叠 |
| 标点 | 中英文标点混用、引号不匹配、顿号逗号混用 |
| 口语残留 | 是否残留了"说白了""搞""挺""蛮""巨"等不适合正式平台的表达（知乎版重点检查） |
| AI 句式 | 是否有过强的公式化反差句、硬转折、刻意制造反转的句式（如"不是……而是……""重点来了""答案来了""真正拉开差距的是……"），小红书版重点检查 |
| 重复表达 | 相邻句子是否存在语义重复、同一词汇短期内多次出现 |
| 语义冗余 | "联合...共建""目的是为了""目前...当前"等句式，两个词表达同一含义，删掉一个 |

**处理方式**：

逐条标注问题句子，给出修改建议，然后输出修正后的全文。格式如下：

```
### 文本质量检视 — 选题 1（知乎版）

**问题清单**：
- [第X句] "xxx" → 语病：缺主语，建议改为 "xxx"
- [第X句] "xxx" → 错别字："的"应为"地"
- [第X句] "xxx" → 通顺度：句式过长，建议拆为两句
- （如无问题，写"未发现问题"）

**修正后正文**：
{修正后的完整正文}
```

**规则**：
- 每条问题必须给出明确的修改建议，不允许只标注"不通顺"而不给方案
- 修正后正文必须是完整可发布的，不允许留"[此处待修改]"等占位符
- 如无任何问题，直接输出"文本质量检视通过"，不强制要求修改

---

### 步骤 5：事实自检报告

对每条生成的短文，输出事实层面的校验结果。

**特别要求：修改也要回源验证**

无论是初稿生成还是后续修改，只要出现以下类型的表述，都必须回到原文核验后才能保留：

- 功能边界句：如某个工具"支持上传、维护、分发"等
- 比较句：如"前者是 X，后者是 Y"、"A 比 B 更……"
- 归纳句：如"这意味着……"、"本质上……"、"可以理解为……"
- 宣传句：如"鼓励开发者共建"、"形成完整闭环"等带总结意味的判断

如果原文没有明确支持，就必须收紧表述，改成原文可直接支撑的版本，不能为了顺畅而补全逻辑。 

输出格式：

```
## 自检报告

### 选题 1：{主题名}

#### 知乎版
| 检查项 | 结果 |
|--------|------|
| 事实可追溯 | ✅ / ⚠️ / ❌ |
| 数据可追溯 | ✅ / ⚠️ / ❌ |
| 案例可追溯 | ✅ / ⚠️ / ❌ |
| 因果关系准确 | ✅ / ⚠️ / ❌ |
| 限定词保留 | ✅ / ⚠️ / ❌ |
| 无新增推断 | ✅ / ⚠️ / ❌ |
| 网址/平台名准确 | ✅ / ⚠️ / ❌ |
| 平台声称可验证 | ✅ / ⚠️ / ❌ |
| 发布来源完整 | ✅ / ⚠️ / ❌ |
| 主体宣传一致 | ✅ / ⚠️ / ❌ |
| 功能边界准确 | ✅ / ⚠️ / ❌ |
| 比较/归纳有依据 | ✅ / ⚠️ / ❌ |

风险句子：
- [第X句] "xxx" — 风险：原文未明确说 Y，建议改为 "xxx" 或人工确认
- （如无风险句子，写"无"）

#### 小红书版
（同上格式）

### 综合评估
- 可直接发布：选题 #N, #M

- 建议人工重审：选题 #Z（附原因）
```

### 步骤 6：最终输出

每个选题输出为一个独立的 markdown 文件，文件内容仅包含可发布的正文，不含任何中间分析过程。

**文件命名**：`{序号}-{简短主题描述}.md`，如 `01-team-skills-发布.md`

**文件内容要求**：
- 最终稿中禁止出现"根据原文""文里提到""原文给出的判断是"等审校腔或元描述，核验动作只发生在后台，不能泄漏到成稿里
- 是"复制即发布"的最终稿，读者看到的就是全部内容
- 中间过程（台账、选题清单、检视报告）单独归档，不混入正文文件

**输出模板**：

```markdown
# {标题}

{正文段落一}

{配图/视频（如有）}

{正文段落二}

...
```

**目录结构建议**：

```
output/
├── evidence-ledger.md        ← 证据台账（中间产物，归档用）
├── zhihu/                    ← 知乎版成稿
│   ├── 01-{主题}.md
│   └── 02-{主题}.md
└── xhs/                      ← 小红书版成稿（含图文 + 视频文案）
    ├── 01-xhs-{主题}.md
    ├── 02-xhs-{主题}.md
    └── cover-xxx.png         ← AI 生成配图
```

## 核心原则

1. **先证据，后表达**：台账未建立，不开始写稿
2. **先单点，后平台化**：选题未确定，不开始改写
3. **先通顺，后校验**：文本质量检视未通过，不进入事实自检
4. **先正确，后传播**：事实校验未通过，不输出最终稿
5. **所有输出可回溯**：每一句话都能在台账中找到对应条目

## 边界情况处理

- **原文内容密度低**：如实告知用户，建议减少拆分数量或换一篇
- **原文专业术语多**：在台账中标注术语，改写时保留术语但补充通俗解释
- **原文立场偏激**：在台账中标注，选题时过滤掉证据不足的极端观点
- **用户要求拆分数量超出内容密度**：告知上限，按实际可拆分数量输出
- **原文为英文**：先翻译为中文再做拆解，台账中保留原文引用
- **原文来自受限平台**（微信公众号等有反爬机制的来源）：优先请用户粘贴正文或导出本地文件；如用户提供了离线 markdown + 图片/视频资源，直接读取本地文件，并遵循媒体资源处理规范嵌入
