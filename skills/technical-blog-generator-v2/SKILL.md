# technical-blog-generator-v2

用于把"指定技术主题"自动检索为多篇社区文章素材，并转化为可信、可读、可发布的小红书技术推文。

v2 相比 v1 的改进：
- 引入结构化证据台账（参考 article-atomize 的证据驱动理念）
- 新增关键词清单与 CTA 提取
- 完整的小红书写作规范（标题、正文、文风、标签、宣传链接）
- 实践/案例类稿件的展示效果要求
- 新增文本质量检视与事实自检步骤
- 规范结构类帖子遵循原文叙事逻辑的规则

核心原则：
- 不捏造事实、不编造代码、不虚构收益数据。
- 风格要有吸引力，但不过度营销，不夸张承诺。
- 先产出草稿并询问是否发布；未得到明确同意前，禁止执行发布动作。

## 适用场景

- 用户给出技术主题，希望自动检索并总结为小红书风格内容。
- 来源文章涉及实现细节，且需要结合源码验证结论。
- 用户希望在最终确认后，通过浏览器能力发布一条小红书推文。

## 输入要求

开始执行前，至少收集以下信息；缺失时先追问：
1. 技术主题（必填，例如"RAG 检索优化""Python 异步任务调度"）
2. 目标读者（如后端开发、前端工程师、AI 应用开发者）
3. 是否需要发布（默认否；即使用户说"要发布"，也要在最终发布前再确认一次）

可选输入：
- 用户指定参考链接（如有则优先纳入分析）
- 期望字数（默认 600-900 中文字）
- 语气偏好（理性拆解/经验复盘/踩坑避坑）
- 是否强制包含原理图（默认是）

## 执行流程（必须按顺序）

### 第 1 步：基于主题检索与证据台账

1. 基于用户给定主题，使用搜索与网页抓取工具检索候选文章（优先近一年、内容完整、技术细节充分的来源）。
2. 至少筛选 2-5 篇高相关内容进行阅读与交叉核验；若用户给了参考链接，则必须纳入。
3. 抽取以下事实并形成结构化"证据台账"：

```
## 证据台账

### 1. 核心论点
- 论点：xxx
- 来源：[文章标题 + URL] 第X段 "原句摘录..."
- 置信度：高/中/低

### 2. 支撑论据
- 论据：xxx
- 来源：[文章标题 + URL] 第X段 "原句摘录..."
- 类型：事实/数据/案例
- 置信度：高/中/低

### 3. 关键数据
- 数据：xxx
- 来源：[文章标题 + URL] 第X段 "原句摘录..."
- 置信度：高/中

### 4. 不确定表述
- "xxx"（来源：[文章标题 + URL] 第X段）— 作者使用了"可能""有些情况下"等限定词，改写时必须保留

### 5. 关键词清单
从来源文章中提取以下三类关键词，后续标题、正文、标签必须覆盖：

产品/项目名：涉及的具名产品、项目、组织名
核心概念/范式：提出的专有概念或方法论
关键平台/资源：给出的独立网址或平台名

### 6. 行动号召识别
从来源文章中识别所有推广性质的呼吁（CTA），单独列出：
- 号召内容：xxx
- 来源：[文章标题 + URL] 第X段
- 类型：下载使用 / 参与共建 / 上传分享 / 关注社区 / 其他
```

**规则**：
- 每条台账必须附原文引用，不允许凭印象归纳
- 置信度标准：高=原文明确陈述，中=原文暗示或可合理推断，低=仅有碎片信息
- 不确定表述必须单独列出，改写时作为约束条件
- 若文章带有明显宣传/发布属性，在拆解阶段必须同步抽取"发布来源/发布主体/配套发布内容"三项信息
- 对工具、平台、能力的职责边界要单独记录，避免在改写时把多个能力顺滑地合并成一个功能包

若检索结果信息不足，必须明确标注"来源未给出"，不要补写主观猜测。

### 第 2 步：判断是否需要源码验证

出现以下任一情况时，必须下载源码辅助分析：
- 原文包含具体实现描述，但表述模糊或可能有歧义。
- 需要给出"如何使用/如何启动/demo case"，但原文仅有结论无步骤。
- 用户明确要求"结合代码分析"。

源码验证要求：
1. 使用 `bash` 下载仓库（如 `git clone`）并定位关键文件。
2. 使用 `read_file` 阅读关键代码/README，提炼可被验证的结论。
3. 在证据台账中补充源码依据，写明"依据了哪些文件/命令结果"。

禁止在未查看源码时写出具体函数名、配置项、启动参数等细节。

### 第 3 步：生成小红书草稿（不发布）

在产出正文前，先读取模板（`templates/xiaohongshu_post_template.md`），按模板的节结构和风格约束填充。

模板结构：
- 标题：≤20 字，单句完整，有吸引力但不夸张
- 简要概述：背景 + 方案 + 价值（2-4 句）
- 技术方案：方案介绍 + 原理图（可选）+ 效果收益
- 如何使用 / 如何启动 / demo case
- 结尾行动引导
- 证据清单附录

**小红书标题规则**（参考 article-atomize）：
- 标题必须是一个可单独成立的完整句，不能用"最近了解到一个项目"这类泛开场充当标题
- 如果原文属于发布/上线/开源/推出类内容，标题应显式包含"发布""上线""开源"等事件词，强化发布感
- 标题要直接说清楚发现了什么/差异在哪里/结果有多强，不要把信息留到正文里
- 避免公式化反差句式，如"不是 X 而是 Y"——这种表达读起来偏模板感，能直接陈述就优先直接陈述
- 优先使用"一句话创建""刚发布"这类强信息词，但情绪词必须服务于正文核心机制
- 标题要有信息量但不能像标题党，准确反映内容

**小红书正文规则**：
- 开头：一句话交代这篇只讲一个什么判断或发现。直接给核心认知，不在铺陈中慢慢靠近
- 中段：每段 2-3 句，用 emoji 做视觉分隔。每一段都必须服务于同一条主线
- 结尾：一句回扣开头主旨的 takeaway
- 正文动笔前，先在内部分配一句"一句话主旨"，用它来检验每一段是否服务同一个重心

**小红书文风规则**：
- 采用用户/开发者第一视角，分享感强，像一个发现了好东西忍不住分享的朋友
- 短句为主，单句不超过 25 字
- 口语化但不水，避免"绝绝子""YYDS""太酷啦"等空洞感叹
- 主动回避"不是……而是……""重点来了""真正拉开差距的是……"等公式化转折句——这类句式的 AI 模板感太重
- emoji 用于分段标记，每段 1-2 个，不要连续堆砌
- 技术术语保留，但紧跟一句大白话解释（"即……""简单说就是……"）
- 不做虚假"亲身经历""我朋友""我试了"等编造叙事——如果原文是别人的案例，就说"有人试了""案例里"
- 用具体场景代替抽象描述：不说"效率很高"，说"以前要手动配半天，现在一句话搞定"
- 如果某篇承担的是"结构解析"角色，正文里仍可适度补入"怎么创建/怎么上手"的信息

**实践/案例类稿件额外要求**：
- 除了讲创建过程和协作机制，必须补一段"展示效果"：读者看完需要知道屏幕上具体看到了什么、为什么值得发
- 允许的展示效果（必须回源可证）：
  - 生成出的角色/配置/结构长什么样
  - 运行时过程有哪些节点可见
  - 最终输出的结果形态
  - 与手动操作相比省掉了哪一步
- 禁止只讲机制对不对，不讲最后看到了什么

**宣传链接规则**：
- 如果来源文章带有宣传/发布属性，正文末尾（标签前）应补入宣传链接
- 链接来源：来自台账关键词清单中的关键平台/资源项
- 覆盖策略：每篇至少覆盖一条链接；如果正文未提及 Hub，则同时补 Hub 地址并加一句简短介绍
- 链接必须来自来源文章明确给出的地址，不编造

**话题标签**：
- 3-5 个，格式 `#标签名`，放在正文末尾
- 标签必须覆盖台账关键词清单中的产品名和核心概念，不可遗漏
- 组合策略：产品名 + 核心概念在前（精准搜索），领域泛标签在后（泛流量）

以上规则在写作前一次性读入，写作时逐条对照。

### 第 4 步：文本质量检视与事实自检

草稿产出后，先做两轮自检再进入发布确认。

**文本质量检视**（检查项）：
- 语病：缺主语、缺谓语、搭配不当、句式杂糅
- 错别字：同音错字、形近错字
- 通顺度：句子是否拗口、虚词堆叠
- 标点：中英文标点混用、引号不匹配
- AI 句式：是否有"不是……而是……""重点来了""答案来了"等公式化转折句
- 重复表达：相邻句子语义重复
- 语义冗余："联合...共建""目的是为了"等句式

每条问题必须给出明确的修改建议，修正后正文必须是完整可发布的。

**事实自检**（检查维度）：
- 事实可追溯、数据可追溯、案例可追溯
- 因果关系准确、限定词保留、无新增推断
- 网址/平台名准确、平台声称可验证（如声称"在 GitHub 发现"需先验证该内容确实存在）
- 发布来源完整、功能边界准确、比较/归纳有依据

修正后输出完整自检报告。

### 第 5 步：图片与原理图策略

1. 若来源文章已有可用图：
   - 优先复用来源图，并在文案里标注"图来自来源文章"
   - 路径中的空格用 `%20` 编码
2. 若来源文章无可用图：
   - 生成一段 mermaid 原理图（使用模板 `templates/diagram_mermaid_template.md`）
   - 图只表达关键流程，不堆砌细枝末节

若输出目标是小红书，还需遵循媒体资源处理规范：
- 实践向/案例向帖子默认应有配图，不建议做成纯文字长帖
- 优先策略不是自动抽帧，而是在成稿中写"建议从哪个视频截哪一类画面"
- 如果视频截图效果不理想，再退回到 AI 生图方案

### 第 6 步：发布前二次确认（强制）

生成草稿后，必须明确询问用户：
- "是否现在发布到小红书？"

只有在用户给出明确肯定（如"确认发布/现在发布/去发布"）后，才允许进入发布步骤。

以下情况均视为"未确认"，不得发布：
- 用户未回复
- 用户表达修改意见但未确认最终稿
- 用户只说"先看看/再等等/暂不发布"

### 第 7 步：浏览器发布（仅确认后执行）

确认后才使用 `browser_run_task` 执行小红书发布流程，任务中应包含：
- 使用最终确认版文案
- 处理标题、正文、图片/原理图素材
- 到达发布前的必要校验（内容完整、无明显截断）

若遇到登录态失效、验证码、风控弹窗：
- 立即暂停自动化
- 明确告知用户需要手动接管
- 待用户完成后再继续

## 输出格式要求

默认输出按以下顺序：
1. 证据台账（来源文章/代码依据）
2. 小红书草稿（可直接复制）
3. 原理图（如适用，提供 mermaid 代码）
4. 文本质量检视与事实自检报告
5. 发布确认问题（必须）

## 模板引用

- 正文模板：`templates/xiaohongshu_post_template.md`
- Mermaid 模板：`templates/diagram_mermaid_template.md`

在产出正文前，优先读取模板并按模板填充，确保结构稳定。
